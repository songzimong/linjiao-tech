# 群星体系 · 完善方案 (Starlight System v3.0)

> 版本：v3.0 | 日期：2026-06-22
> 主题：在三层治理基础上增加应急通道、负载均衡、自进化机制

## 一、新增：应急通道 (Emergency Channel)

### 设计原理
三层架构的代价是延迟。紧急情况下，底层可以直接触达顶层，但必须有安全兜底。

### 通道规则

```
蜂群层发现紧急情况
    ↓
1. 玄武实时监护（必须）
    ↓
2. 直接提交方案给集权层（跳过群星层）
    ↓
3. 集权层快速裁决（限时 3 分钟）
    ↓
4. 执行 + 事后复盘
```

### 触发条件
- 系统安全事件（玄武检测到攻击/入侵）
- 时间敏感任务（用户明确要求"立刻"）
- 蜂群层发现重大机会/威胁
- 群星层 Agent 失联或无响应

### 限制
- 每次使用后必须复盘，记录原因
- 滥用（每月 >3 次无理由使用）→ 降低该蜂群规模上限
- 集权层有权否决应急通道的方案

### 安全兜底
玄武在应急通道中拥有**最高权限**——即使集权层来不及反应，玄武也可以单方面终止蜂群层操作。

## 二、新增：负载均衡机制 (Load Balancer)

### 问题
麒麟是集权层唯一决策节点，容易过载。

### 解决方案：分级决策

| 决策级别 | 决策者 | 适用范围 |
|---------|--------|---------|
| L1 - 战略决策 | 麒麟 | 公司方向、资源分配、红线规则 |
| L2 - 战术决策 | 麒麟 + 群星层长老 | 任务路由、Agent 选择、优先级排序 |
| L3 - 执行决策 | 虾兵自主 | 具体执行方式、工具选择、代码实现 |

### 长老机制
从虾兵中选出 3 位"长老"，协助麒麟做 L2 决策：

```
长老候选人条件：
- 历史成功率 ≥ 80%
- 领域 expertise 突出
- 无安全违规记录

长老职责：
- 协助麒麟做任务路由决策
- 在麒麟过载时临时接管 L2 决策
- 定期向麒麟汇报领域进展

长老轮换：每季度评估一次，不合格退出
```

### 当前长老候选人（按领域匹配度）
1. **青龙** — 技术研发（代码能力强，历史成功率高）
2. **貔貅** — 财务管理（细心严谨，无违规记录）
3. **辟邪** — 安全防护（天然适合辅助玄武）

> 注：长老身份由麒麟根据实际表现动态评估，此处为初步建议。

## 三、新增：自进化机制 (Self-Evolution)

### 问题
现有架构是静态的——规则写死了就不会变。但业务在变，Agent 的能力也在变。

### 进化方向

#### 3.1 Agent 能力进化
```
每次任务完成后：
  结果成功 → skill_level += 0.05
  结果失败 → skill_level -= 0.05
  连续成功 5 次 → 获得新 capability 标签
  连续失败 3 次 → 触发能力评估，麒麟重新分配领域
```

#### 3.2 调度策略进化
```
每 7 天复盘一次：
  - 哪些任务类型用错了调度模式？
  - 哪些 Agent 经常被分配不擅长的任务？
  - 蜂群模式的成功率是否高于常规模式？
  
根据复盘结果自动调整调度策略。
```

#### 3.3 三层结构进化
```
当某一层连续 30 天无活动时：
  → 评估该层是否仍有必要存在
  → 必要时合并或裁撤
  
当某一层负载持续 >80% 时：
  → 考虑在该层内部增设子层级
  → 或增加该层的 Agent 数量
```

#### 3.4 知识沉淀进化
```
蜂群涌现的模式 → 固化为规则
规则被验证有效 → 加入可复用规则库
规则连续 10 次不适用 → 标记为"待淘汰"
规则连续 30 天未使用 → 自动归档
```

## 四、新增：绩效评估体系 (Performance Evaluation)

### Agent 个人绩效

```
绩效分数 = 任务完成率 × 40% 
         + 方案质量 × 30% 
         + 协作效率 × 20% 
         + 安全合规 × 10%
```

| 绩效等级 | 分数范围 | 待遇 |
|---------|---------|------|
| S | ≥95 | 优先派活，技能提升奖励 |
| A | 85-94 | 正常派活，季度表彰 |
| B | 70-84 | 正常派活，需改进 |
| C | 60-69 | 减少派活，能力评估 |
| D | <60 | 暂停派活，重新培训 |

### 层级绩效

| 层级 | 评估指标 | 频率 |
|------|---------|------|
| 集权层 | 决策准确率、响应速度、战略方向正确性 | 每周 |
| 群星层 | 任务完成率、领域深耕深度、横向协作次数 | 每月 |
| 蜂群层 | 涌现方案质量、共识达成效率、创新次数 | 每次蜂群结束后 |

## 五、新增：容灾机制 (Disaster Recovery)

### 麒麟失效
```
如果麒麟连续 2 小时无响应：
  1. 长老机制启动，3 位长老投票决定紧急事务
  2. 玄武接管安全审计
  3. 虾兵进入待机模式，不执行新任务
  4. 用户可通过 WebChat 直接联系长老
```

### 白泽失效
```
如果白泽记忆系统不可用：
  1. 切换到只读模式（可读取 MEMORY.md 备份）
  2. 新记忆暂存到临时目录
  3. 恢复后自动同步
  4. 期间虾兵不执行需要历史上下文的复杂任务
```

### 玄武失效
```
如果安全审计系统不可用：
  1. 所有 Agent 进入保守模式（只读操作）
  2. 禁止蜂群模式
  3. 所有任务需用户手动确认
  4. 恢复后恢复正常模式
```

### 全链路失效
```
如果超过 2 个层级同时失效：
  1. 系统进入维护模式
  2. 通知用户
  3. 等待人工恢复
```

## 七、v2.0 → v3.0 改进总结

| 维度 | v2.0 | v3.0 |
|------|------|------|
| 决策层级 | 麒麟独揽 | 麒麟 + 长老分级 |
| 应急响应 | 无 | 应急通道 + 长老接管 |
| 负载均衡 | 无 | 三层决策 + 长老协助 |
| 自进化 | 无 | Agent/策略/结构/知识四维进化 |
| 绩效评估 | 无 | 个人 + 层级双维度评估 |
| 容灾机制 | 无 | 单/双/全链路三级容灾 |
| 安全兜底 | 玄武事后审查 | 玄武实时监听 + 单边终止权 |
