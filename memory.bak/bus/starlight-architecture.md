# 群星体系 · 完整架构 (Starlight System)

> 版本：v1.0 | 日期：2026-06-22
> 融合：麒麟集中调度 + 蜂群思维涌现 = 群星体系

## 一、核心哲学

**有约束的涌现** — 蜂群不是随机碰撞，而是在安全边界内、质量可评、结果可学的智能涌现。

### 三层架构

```
┌─────────────────────────────────────────────────┐
│                   麒麟调度层                      │
│  意图解析 → 模式选择 → 蜂群编排 → 结果汇总        │
├─────────────────────────────────────────────────┤
│                  蜂群涌现层                       │
│  Agent ↔ Agent 网状对话 → 方案碰撞 → 共识涌现      │
├─────────────────────────────────────────────────┤
│                  安全护栏层                       │
│  玄武实时监听 → 质量评估 → 知识沉淀               │
└─────────────────────────────────────────────────┘
```

## 二、三种调度模式

### 模式 1: 常规模式 (星型调度)
```
麒麟 → Agent A → 麒麟 → 用户
```
- 适用：任务执行、代码交付、日常运维
- 特点：高效、可控、可预测

### 模式 2: 蜂群模式 (网状涌现)
```
麒麟 → [Agent A ↔ Agent B ↔ Agent C] → 麒麟汇总 → 用户
```
- 适用：创意 brainstorm、策略推演、多方案对比
- 特点：发散、涌现、高质量共识

### 模式 3: 混合模式 (分层调度)
```
麒麟 → 蜂群讨论 → 麒麟筛选 → 执行 → 麒麟汇总
```
- 适用：复杂决策、跨域任务
- 特点：兼顾发散与收敛

## 三、蜂群引擎核心机制

### 3.1 动态蜂群规模

| 议题复杂度 | Agent 数量 | 对话轮次上限 | 超时时间 |
|-----------|-----------|------------|---------|
| 简单 | 3-5 | 10 | 3 分钟 |
| 中等 | 8-12 | 20 | 5 分钟 |
| 复杂 | 全量 | 50 | 10 分钟 |
| 极复杂 | 分层蜂群 | 嵌套 | 15 分钟 |

复杂度由麒麟根据议题自动评估。

### 3.2 质量加权投票

每个 Agent 的权重 = base_weight × domain_bonus × history_factor

- **base_weight**: 基础权重（所有 Agent = 1.0）
- **domain_bonus**: 领域加成（该 Agent 在此领域的 skill_level / 5）
- **history_factor**: 历史成功率（过去 10 次同类任务的成功率）

投票得分 = Σ(方案得分 × 权重)

### 3.3 过程安全护栏

玄武在蜂群对话中**实时监听**：
- 敏感关键词触发 → 暂停蜂群 → 人工审查
- 异常对话模式 → 标记 → 事后分析
- 越界行为 → 立即终止该 Agent 参与

### 3.4 知识沉淀系统

每次蜂群结束后自动执行：

```json
{
  "consensus_solution": "最优方案（写入记忆宫殿）",
  "failed_approaches": [
    {"approach": "方案A", "reason": "失败原因"},
    {"approach": "方案B", "reason": "失败原因"}
  ],
  "emerged_patterns": "涌现出的新模式/规则",
  "agent_contributions": {
    "AgentA": {"votes": 3, "weight": 1.5, "impact": "高"},
    "AgentB": {"votes": 1, "weight": 0.8, "impact": "低"}
  }
}
```

### 3.5 递归蜂群引擎

复杂任务支持嵌套蜂群：

```
Level 0: 麒麟发布任务
  ↓
Level 1: 蜂群 A（方向讨论）
  → 选出 3 个候选方向
  ↓
Level 2: 蜂群 B（方案细化）
  → 对每个方向生成 2 个方案
  → 共 6 个方案
  ↓
Level 3: 蜂群 C（最终评审）
  → 6 个方案质量加权投票
  → 选出最优
  ↓
Level 0: 麒麟执行最优方案
```

递归深度上限：3 层（防止无限嵌套）

### 3.6 冷启动信任度评估

新 Agent 或低频 Agent 的初始权重 = 0.5（半权）

建立信任度的方法：
1. **交叉验证**：与其他 Agent 的方案对比一致性
2. **麒麟评估**：人工打分后提升权重
3. **快速迭代**：连续 3 次高质量输出后，权重恢复至正常

## 四、蜂群协议

### 4.1 消息格式（蜂群专用）

```json
{
  "from": "agent_id",
  "type": "propose|critique|support|vote|consensus",
  "content": "<方案/意见/投票>",
  "score": 0.0,          // 自评分数 0-10
  "confidence": 0.0,     // 自信度 0-1
  "references": ["相关记忆文件"],
  "round": 1,            // 当前轮次
  "timestamp": "<ISO-8601>"
}
```

### 4.2 蜂群生命周期

```
INIT → DISCUSS → VOTE → CONSENSUS → ARCHIVE
  ↑                              ↓
  └──── TIMEOUT / ABORT ─────────┘
```

### 4.3 共识判定条件

满足以下任一条件即达成共识：
1. 某方案加权得分 ≥ 总权重的 60%
2. 连续 3 轮无新方案提出
3. 所有 Agent 投票一致

## 五、与现有体系的集成

### 5.1 与白泽记忆集成
- 蜂群结果自动写入对应殿堂
- 历史蜂群记录供未来参考
- 涌现模式提炼为可复用规则

### 5.2 与玄武安全集成
- 实时监听模式（蜂群进行中）
- 事后审查模式（蜂群结束后）
- 一票否决权不变

### 5.3 与虾兵集成
- 所有虾兵均可参与蜂群
- 能力画像决定 domain_bonus
- 负载状态决定是否可参与

### 5.4 与任务编排器集成
- 蜂群作为可选调度策略
- 在 AGENTS.md 中新增蜂群触发条件
- 麒麟自动判断何时使用蜂群
